# IEC 104 文档优化 PRD

## 背景

IEC 104 数采平台有两份核心文档：
- `docs/iec104.md` (341行) - 平台概述、架构、配置
- `docs/iec104-protocol.md` (1160行) - 消息格式、信息体、控制命令

当前存在内容重复、结构不清晰等问题，需要优化。

## 问题清单

### 1. 内容重复
- `device_point_mapping` 表结构在两个文件中都有定义
  - `iec104.md:227-247` (简化版，PostgreSQL)
  - `iec104-protocol.md:138-163` (完整版，SQLite，含 description 字段)

### 2. 结构问题
- `iec104-protocol.md` 过长(1160行)，包含两大独立主题：
  - 消息格式/信息体（监视方向）
  - 控制命令接口（控制方向）
- 两个文件都缺少完整目录导航

### 3. 内容边界模糊
- `iec104.md` 第6节"ASDU类型支持"只有一句话指向 protocol.md
- `iec104.md` 的 TDengine 表结构与 protocol.md 的消息格式关联不强

## 优化方案

### 方案 A：拆分为三个文件（推荐）

```
docs/
├── iec104.md              # 平台概述（精简）
├── iec104-message.md      # 消息格式与信息体（监视方向）
└── iec104-command.md      # 控制命令接口（控制方向）
```

**文件职责**：

| 文件 | 内容 | 预估行数 |
|------|------|----------|
| `iec104.md` | 概述、架构、数据流、服务详解、弱校验、点位管理、TDengine、MQTT Topic | ~300 |
| `iec104-message.md` | 消息格式、PointMapping、ASDU映射、信息体结构、附录、消费指南 | ~600 |
| `iec104-command.md` | 接入方式、7个带类型接口、通用接口、其他接口、响应机制、错误码、TypeId对照表 | ~500 |

**优点**：
- 单文件长度可控，便于阅读和维护
- 监视/控制两个方向职责清晰
- 消费者可按需查阅

**缺点**：
- 需要更新所有交叉引用链接

### 方案 B：保持两文件，去重+优化结构

保持现有两个文件，但：
1. 删除 `iec104.md` 中重复的表结构，改为引用
2. 为 `iec104-protocol.md` 添加完整目录
3. 在 `iec104-protocol.md` 内部用分隔符区分监视/控制

**优点**：改动小
**缺点**：protocol.md 仍然过长

## 验收标准

1. 无重复内容（表结构、接口定义等）
2. 每个文件有完整目录
3. 交叉引用链接正确
4. 内容完整，无遗漏
5. 行业术语一致
