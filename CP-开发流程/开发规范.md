# 敏捷开发工作流规范

本文档定义 `CP-开发流程/` 目录下的精简敏捷开发工作流、文件组织规范与 AI 团队协作方式。
核心目标：**开启开发先执行 `trellis:start`，确保使用 AI 团队和 Trellis；老板需求统一进入 `Backlog.md`；AI 按需载入上下文和专业大模型能力，避免一次性加载过多 token。**

---

## 一、启动原则

### 1.1 开发启动必须使用 Trellis

当用户表达「开始开发 / 执行 Sprint / 做 Backlog / 开启开发 / 接着干」等开发意图时，AI 必须先进入 Trellis 启动流程：

```text
trellis:start
  → 识别开发者身份、当前任务、项目上下文
  → 检查 git 状态和工作区风险
  → 按需读取 .trellis/spec/ 规范索引
  → 进入 AI 团队调度流程
```

在 Trae 环境中，如果无法直接执行命令形式的 `trellis:start`，等价执行项目内启动工作流：

```bash
python3 ./.trellis/scripts/get_context.py
```

以 `trellis:start` 或等价的 Trellis 上下文读取作为启动动作。`.agent/workflows/**` 属于工具自带/托管目录，只作为运行环境已有能力使用，不纳入 CP 流程自定义修改范围。

### 1.2 必须使用 AI 团队

CP 开发流程默认由 `agile-dev-manager` 作为总指挥，自动调度 AI 团队角色：

| 阶段 | 默认角色 | 职责 |
| --- | --- | --- |
| Planning | PM | 梳理 Backlog、拆解 Story、规划 Sprint |
| Execute | Backend / Frontend | 按任务实现代码 |
| Review | QA | 构建、测试、规范检查 |
| Retro | PM | 更新 Backlog、追加变更记录、交付报告 |
| Deploy | DevOps | 按需部署 |

`ai-team` 是角色详细参考文档，只有当角色边界、交接协议或权限冲突不清楚时才读取，不默认整篇加载。

### 1.3 最小上下文原则

AI 不允许为了“保险”一次性读取所有流程文档。读取顺序如下：

| 时机 | 必读 | 按需读取 |
| --- | --- | --- |
| 会话启动 | `trellis:start` 输出 / `.trellis/.current-task` | `.trellis/workspace` 最近摘要 |
| Sprint Planning | `Backlog.md` | `任务清单.md` 当前 Sprint 摘要 |
| 技术方案 / 编码 | 当前 Task、目标代码文件、相关 spec 索引 | `开发计划.md` 对应章节、具体 `.trellis/spec/*` |
| Review | 变更文件列表、当前 Task 验收标准 | 对应规范 Quality Check 章节 |
| Retro | Backlog 当前条目、任务完成状态 | `变更记录.md` 最近 Sprint |

禁止默认全文读取：历史 Sprint、完整变更记录、完整开发计划、完整 ai-team 文档、所有 spec 文件。

## 二、需求输入模式

### 2.1 唯一入口

老板所有需求统一写入项目目录的 `Backlog.md`：

```text
CP-开发流程/{项目名}/Backlog.md
```

不再默认创建或维护 `需求输入.md`。旧项目如存在 `需求输入.md`，只在用户明确要求或迁移旧需求时读取。

### 2.2 Backlog 输入分层

`Backlog.md` 承载三类信息：

| 区域 | 写入方 | 用途 |
| --- | --- | --- |
| 老板输入区 | 老板 | 自由写需求、想法、问题、链接、接口草稿 |
| 产品待办 | AI | 将老板输入梳理为 Story，维护优先级和状态 |
| 当前 Sprint 摘要 | AI | 记录本轮目标、关联 Backlog、状态 |

### 2.3 AI 处理规则

1. 只扫描 `老板输入区` 中未处理条目和 `产品待办` 中待开发/开发中条目。
2. 老板输入处理后，将原条目标记为 `[已梳理 → B-XXX]`，不删除原文。
3. 模糊需求最多问 3 个关键问题；能合理假设的内容直接记录假设并继续。
4. 大段需求不再搬运到多个文件；只提炼关键目标、验收标准、约束进入产品待办。
5. 外部链接只记录引用，不把全文复制进 Backlog；需要时再按需打开。

## 三、术语与编号

| 术语 | 说明 |
| --- | --- |
| Sprint | 一个开发迭代，编号为 S1、S2、S10，自然递增 |
| Backlog | 产品待办，所有需求的唯一入口 |
| Epic | 大粒度业务目标，不直接执行 |
| Story | 可交付的功能切片 |
| Task | AI 可直接执行的最小技术任务 |
| DoD | 完成定义，任务完成前必须满足的检查条件 |

编号规范：

| 类型 | 格式 | 示例 |
| --- | --- | --- |
| Sprint | S{数字} | S1、S12 |
| Task | S{数字}-{两位序号} | S1-01、S12-03 |
| Backlog | B-{三位自增} | B-001、B-128 |

## 四、文件职责

每个项目目录默认包含 4 个核心文件和 1 个说明文件：

```text
CP-开发流程/{项目名}/
├── Backlog.md       # 唯一需求入口 + 产品待办 + 当前 Sprint 摘要
├── 任务清单.md      # 当前 Sprint 执行清单
├── 开发计划.md      # 慢变架构设计，按需读取/更新
├── 变更记录.md      # Sprint 级交付摘要
└── README.md        # 项目流程说明
```

### 4.1 Backlog.md

| 属性 | 说明 |
| --- | --- |
| 定位 | 唯一需求入口 |
| 必读时机 | Sprint Planning |
| 写入规则 | 老板写输入区；AI 梳理产品待办和 Sprint 摘要 |

### 4.2 任务清单.md

| 属性 | 说明 |
| --- | --- |
| 定位 | 当前 Sprint 执行清单 |
| 必读时机 | Execute / Review |
| 写入规则 | AI 边做边更新任务状态，默认只保留当前 Sprint |

### 4.3 开发计划.md

| 属性 | 说明 |
| --- | --- |
| 定位 | 慢变架构设计 |
| 读取时机 | 涉及架构、接口契约、数据模型、关键依赖、系统边界时 |
| 写入规则 | 仅重大架构或接口契约变化时更新 |

### 4.4 变更记录.md

| 属性 | 说明 |
| --- | --- |
| 定位 | Sprint 级交付摘要 |
| 读取时机 | Retro 或需要追溯最近交付时 |
| 写入规则 | Sprint 完成后追加摘要，不写逐任务流水账 |

## 五、开发模式

### 5.1 新项目启动

```text
trellis:start
  → 激活 agile-dev-manager，确认 AI 团队调度
  → 复制 template 到 CP-开发流程/{项目名}/
  → 老板在 Backlog.md 老板输入区写目标和第一批需求
  → PM 梳理产品待办和 MVP 路线图
  → 仅在需要架构边界时补充 开发计划.md
  → 规划 S1 到 任务清单.md
  → Backend/Frontend 执行
  → QA 验证
  → PM 追加 变更记录.md 并交付报告
```

### 5.2 日常迭代

```text
trellis:start
  → 读取 Backlog.md 未处理输入和待开发条目
  → PM 选择当前 Sprint 范围
  → 写入 / 更新 任务清单.md
  → 按 Task 定位代码和最小规范
  → 执行、验证、回填状态
  → 追加 Sprint 级变更记录
```

### 5.3 小修 / Bugfix

```text
trellis:start
  → 明确问题和目标代码范围
  → 如已有 Backlog 关联则读取对应条目，否则直接定位代码
  → 修复并验证
  → 如属于 CP 项目任务，再回填 Backlog / 任务清单 / 变更记录
```

## 六、按需载入大模型能力

技能和大模型能力按场景触发，不预加载：

| 场景 | 触发能力 |
| --- | --- |
| Sprint / Backlog / 任务拆解 | `agile-dev-manager` |
| 角色边界、交接协议不清楚 | 按需读取 `ai-team` 相关小节 |
| go-zero API/RPC/Model/中间件 | `zero-skills` |
| Eino 编排、Agent、A2UI | `eino-skills` / `eino-learning` |
| 部署模块 | `module-deploy` |
| 复杂技术方案 | 计划模式或 `writing-plans` |
| 需求创意探索 | `brainstorming`，但只围绕当前需求，不扩散 |

原则：先用当前上下文判断是否需要专业能力；需要时只载入相关能力，不全文载入所有技能文档。

## 七、DoD

任务标记为完成前必须满足：

1. 代码实现完成并符合项目规范。
2. `.api` / `.proto` 变更后已执行 `gen.sh`，禁止跳过生成步骤。
3. 项目可用的 build / test / lint / typecheck 已执行。
4. 工具类函数有必要单测。
5. `任务清单.md` 状态已更新。
6. Sprint 完成时已追加 `变更记录.md`。

验证命令必须真实记录；未执行必须说明原因。

## 八、AI 协作流程

### 8.1 Planning

1. 确认已执行 `trellis:start`。
2. 激活 PM 角色。
3. 读取 `Backlog.md` 未处理输入和待开发条目。
4. 必要时最多提出 3 个关键问题。
5. 输出当前 Sprint 目标和任务清单。

### 8.2 Execute

1. 激活 Backend / Frontend 角色。
2. 读取当前 Task 和目标代码范围。
3. 按需读取 `开发计划.md` 相关章节和 `.trellis/spec/` 具体规范。
4. 编码实现并即时验证。
5. 完成后更新 `任务清单.md`。

### 8.3 Review

1. 激活 QA 角色。
2. 根据变更文件读取对应规范。
3. 执行可用验证命令。
4. 输出 PASS / FAIL / CONDITIONAL PASS。

### 8.4 Retro

1. 激活 PM 角色。
2. 更新 Backlog 状态。
3. 追加 `变更记录.md` 的 Sprint 级摘要。
4. 输出交付报告。

---

## 九、新项目初始化检查清单

1. 执行 `trellis:start` 或等价启动工作流。
2. 确认使用 `agile-dev-manager` 管理 Sprint，并由 AI 团队自动调度角色。
3. 从 `template/` 复制文件到新项目目录。
4. 让老板只编辑 `Backlog.md` 的老板输入区。
5. AI 梳理产品待办和当前 Sprint。
6. 仅在需要架构设计时维护 `开发计划.md`。
7. 执行、验证、交付并追加 `变更记录.md`。
