# 敏捷开发工作流规范

本文档定义 `CP-开发流程/` 目录下的敏捷开发工作流、文件组织规范与 AI 协作流程。
所有项目目录必须遵循此规范，形成统一的「计划 → 执行 → 回填 → 回顾」闭环。

---

## 一、敏捷开发核心概念

### 1.1 术语定义

| 术语 | 英文 | 说明 |
| --- | --- | --- |
| Sprint | Sprint (Scrum Guide) | 一个固定周期的开发迭代，编号格式为 S01、S02、S03... |
| Backlog | Product Backlog (Scrum Guide) | 所有待实现的需求池，按优先级排序，持续维护 |
| Epic | Epic (SAFe / PMI-ACP) | 大粒度的业务目标或功能主题，不可直接执行 |
| Story | User Story (PMI-ACP / XP) | 从用户视角描述的需求单元，可在 1~2 个 Sprint 内完成 |
| Task | Task (Scrum Guide) | Story 的技术拆解，AI 可直接执行的最小工作单元 |
| DoD | Definition of Done (Scrum Guide) | 一项任务被标记为「完成」所必须满足的验收条件 |
| 便签 | Boss Scratchpad | Backlog 顶部的自由格式区域，老板随意写入粗糙想法，PM 自动梳理 |
| 里程碑 | Milestone (PMI-ACP) | 项目阶段划分，通常为 MVP → 完整版 → 增强版三级 |
| 依赖分析 | Dependency Analysis | 识别模块间的依赖层级：地基模块、核心业务模块、锦上添花模块 |
| 进度检测 | Progress Detection (SpecForge) | 扫描项目代码，自动标记已存在模块为已完成 |

### 1.2 需求拆解层级

```
Epic（史诗 / Epic）
  └── Story（用户故事 / User Story）
        └── Task（任务 / Task）
```

- **Epic**: 对应一个完整的业务目标，如「DJI 上云前置服务」「trigger 功能开发」
- **Story**: 对应一个可交付的功能切片，如「实现设备注册接口」「支持 WebSocket 推送」
- **Task**: 对应一个具体的技术动作，如「定义 proto 文件」「编写 handler 逻辑」「编写单元测试」

### 1.3 完成定义（Definition of Done, DoD）

一个 Task 标记为完成前，必须满足以下全部条件：

1. 代码实现完毕，符合项目编码规范
2. `go build` 编译通过，无错误
3. 工具类函数附带单元测试且测试通过
4. 变更记录已回填到 `变更记录.md`
5. 任务清单状态已更新

## 二、开发模式

### 2.1 首次开发模式（First Development / Project Kickoff）

适用场景：项目从零开始，用户提供项目大纲或初始构想。

```
用户提供项目大纲（可能来自其他 AI 或自行编写）
        ↓
AI 读取大纲，润色补充技术细节
        ↓
AI 产出 → 开发计划.md（核心设计文档）
        ↓
AI 从开发计划中拆分第一个 Sprint 的任务
        ↓
AI 写入 → Backlog.md（完整待办列表）
AI 写入 → 任务清单.md（S01 任务）
        ↓
AI 执行 Sprint S01
        ↓
AI 回填 → 变更记录.md + 任务清单.md
```

**流程要点**:

- 用户大纲可以是粗略的，AI 负责润色、补充边界条件和技术方案
- PM 首次启动时执行依赖分析（地基/核心/锦上添花）和里程碑规划（MVP/完整版/增强版），产出开发路线图
- `开发计划.md` 是首次产出的核心文档，后续迭代中仅在用户要求时修改
- 首个 Sprint 应优先覆盖基础架构和核心链路（对应 Milestone 1: MVP）

### 2.2 迭代开发模式（Iteration / Sprint Cycle）

适用场景：项目已有基础，用户持续追加需求。

```
用户在 Backlog.md 写入新需求（简要描述即可）
        ↓
AI 读取 Backlog.md，润色需求描述
        ↓
AI 评估优先级（MoSCoW），排列顺序
        ↓
AI 从 Backlog 中选取本 Sprint 范围
        ↓
AI 拆解为 Task → 写入 任务清单.md（新 Sprint 分组）
        ↓
AI 逐项执行（编码 → 测试 → 验证）
        ↓
AI 回填 → 变更记录.md + 任务清单.md
        ↓
AI 更新 → Backlog.md（标记已完成条目）
```

**流程要点**:

- 用户只需在 `Backlog.md` 写入简要需求，无需关注格式
- AI 负责润色、评估优先级、拆解任务
- 每个 Sprint 结束后，AI 主动回顾并更新 Backlog 状态

## 三、目录结构

按项目或功能名称组织目录，根目录为 `CP-开发流程/`。

```
CP-开发流程/
├── 开发规范.md                    # 本文件，全局共享的工作流规范
├── template/                      # 模板目录，新项目从此复制
│   ├── 开发计划.md
│   ├── Backlog.md
│   ├── 任务清单.md
│   └── 变更记录.md
├── DJI上云前置服务/               # 项目目录，按业务名称命名
│   ├── 开发计划.md
│   ├── Backlog.md
│   ├── 任务清单.md
│   ├── 任务清单-历史归档.md       # 可选，历史 Sprint 归档
│   └── 变更记录.md
└── trigger功能开发/               # 另一个项目目录
    ├── 开发计划.md
    ├── Backlog.md
    ├── 任务清单.md
    └── 变更记录.md
```

**命名规则**:

- 目录名使用项目/功能的业务名称，不使用版本号
- 文件名使用中文，`.md` 格式
- 一个 Epic 对应一个项目目录

## 四、文件职责

每个项目目录包含 4 个核心文件：

### 4.1 开发计划.md -- 核心设计文档

| 属性 | 说明 |
| --- | --- |
| 定位 | 项目的顶层设计文档，描述「做什么」和「怎么做」 |
| 产出时机 | 首次开发模式下由 AI 润色用户大纲产出 |
| 修改规则 | 除非用户明确要求，否则不修改核心设计部分 |

**必须包含的章节**:

- 项目目标与边界
- 系统架构设计
- 接口设计（API / gRPC）
- 技术选型与依赖
- 非功能性要求（性能、安全等）

### 4.2 Backlog.md -- 产品待办列表

| 属性 | 说明 |
| --- | --- |
| 定位 | 需求池，替代旧的「下一步计划.md」 |
| 写入方 | 老板在便签区随意写想法，或在产品待办表追加需求；AI 润色并管理优先级 |
| 修改规则 | 老板可随时追加；AI 负责便签梳理、需求澄清、润色、排序、标记状态 |

**便签区域说明**：Backlog.md 顶部设有「老板便签」区域，老板可以任意格式写入粗糙想法。PM 在 Sprint Planning 时自动读取便签，通过苏格拉底式需求澄清（WHY/WHO/WHAT/HOW）引导补充信息，结构化为 Epic → Story 后写入产品待办表，便签原文标记「已梳理 → B-XXX」。

**格式规范**:

```markdown
## 产品待办列表

### Epic: xxx功能

| 编号 | Story | 优先级 | 状态 | Sprint | 备注 |
| --- | --- | --- | --- | --- | --- |
| B-001 | 实现设备注册接口 | Must | 已完成 | S01 | - |
| B-002 | 支持批量导入设备 | Should | 待开发 | - | 用户 04/20 提出 |
| B-003 | 导出设备列表为 CSV | Could | 待开发 | - | - |
```

**优先级标注** -- 使用 MoSCoW 方法：

| 级别 | 含义 | 说明 |
| --- | --- | --- |
| Must | Must Have (this iteration) | 缺少则项目无法交付，当前 Sprint 必须完成 |
| Should | Should Have (next iteration) | 重要但非阻塞，尽量在近期 Sprint 安排 |
| Could | Could Have (nice to have) | 锦上添花，资源充裕时安排 |
| Won't | Won't Have (out of scope) | 明确排除出当前范围，记录但不安排 |

**待办状态**:

| 状态 | 说明 |
| --- | --- |
| 待开发 | 已录入，尚未纳入任何 Sprint |
| 开发中 | 已纳入某个 Sprint，正在执行 |
| 已完成 | 对应 Sprint 全部 Task 完成 |
| 已搁置 | 需求暂时搁置，附注原因 |

### 4.3 任务清单.md -- Sprint 任务跟踪

| 属性 | 说明 |
| --- | --- |
| 定位 | 按 Sprint 分组的任务执行跟踪表 |
| 写入方 | AI 从 Backlog 拆解产出 |
| 修改规则 | AI 边做边更新状态，做完立即标记 |

**格式规范**:

```markdown
## Sprint S02 -- 批量操作与数据校验

...

---

## Sprint S01 -- xxxx功能基础搭建

**Sprint 目标**: 完成核心接口定义与基础框架搭建
**关联 Story**: B-001, B-002

| 编号 | Task | 关联 | 状态 | 说明 |
| --- | --- | --- | --- | --- |
| S01-01 | 定义 proto 文件 | B-001 | ✅ 完成 | - |
| S01-02 | 生成 gRPC 代码 | B-001 | ✅ 完成 | 执行 gen.sh |
| S01-03 | 编写 handler 逻辑 | B-001 | 🚧 进行中 | - |
| S01-04 | 编写单元测试 | B-001 | ⬜ 未开始 | - |
```

**任务编号格式**: `{Sprint编号}-{序号}`，如 `S01-01`、`S02-03`

**任务状态**:

| 标记 | 状态 | 说明 |
| --- | --- | --- |
| ⬜ | 未开始 | 已拆解，尚未执行 |
| 🚧 | 进行中 | AI 正在执行 |
| ✅ | 完成 | 满足 DoD，已回填变更记录 |
| ❌ | 阻塞 | 遇到阻塞问题，附注原因 |

### 4.4 变更记录.md -- 已完成变更记录

| 属性 | 说明 |
| --- | --- |
| 定位 | 按 Sprint 分组的变更归档文档 |
| 写入方 | AI 在任务完成后回填 |
| 修改规则 | 只追加不修改，按 Sprint 倒序排列（最新在前） |

**格式规范**:

```markdown
## Sprint S02 -- 批量操作与数据校验

### S02-01: 实现批量导入接口
- **变更概述**: 新增批量导入设备的 gRPC 接口
- **变更文件**:
  - `app/device/rpc/pb/device.proto` -- 新增 BatchImport 方法
  - `app/device/rpc/internal/logic/batchimportlogic.go` -- 实现批量导入逻辑
- **影响范围**: device 服务 RPC 层
- **验证结果**: go build 通过，单元测试通过

---

## Sprint S01 -- 基础搭建

### S01-01: 定义 proto 文件
...
```

## 五、任务清单归档策略

当 `任务清单.md` 中已完成的 Sprint 数量超过 3 个时，执行归档操作：

### 归档规则

1. **触发条件**: 已完成的 Sprint >= 4 个
2. **保留范围**: 当前文件只保留最近 3 个 Sprint（含进行中的）
3. **归档目标**: 将历史 Sprint 移动到 `任务清单-历史归档.md`
4. **归档格式**: 与原文件格式一致，按 Sprint 倒序排列

### 归档流程

```
检查 任务清单.md 中已完成的 Sprint 数量
        ↓
若 >= 4 个，将最早的 Sprint 追加到 任务清单-历史归档.md
        ↓
从 任务清单.md 中移除已归档的 Sprint
        ↓
确保当前文件只保留最近 3 个 Sprint
```

### 示例

归档前 `任务清单.md` 包含 S01~S05：

- S01、S02 已完成 → 归档到 `任务清单-历史归档.md`
- S03、S04、S05 保留在 `任务清单.md`

## 六、AI 协作流程

### 6.1 运作模式

```
老板（用户）= 提需求、审批、监工、拍板
AI 团队     = 自动闭环执行，主动汇报进展
```

AI 团队包含六个角色，在 Sprint 流程中**自动调度**，无需老板手动切换：

| Sprint 阶段 | 自动激活角色 |
| --- | --- |
| Planning | PM（需求分析）+ Scrum Master（规划） |
| Execute | Backend Developer / Frontend Developer |
| Review | QA Engineer |
| Retro | Scrum Master |
| Deploy（按需） | DevOps Engineer |

### 6.2 Sprint 循环

每个 Sprint 遵循四阶段循环：

```
┌─────────────────────────────────────────────────┐
│                Sprint 循环                       │
│                                                  │
│   Planning ──→ Execute ──→ Review ──→ Retro     │
│   [PM]         [Backend]   [QA]      [Scrum]    │
│      ↑                                   │       │
│      └───────────────────────────────────┘       │
└─────────────────────────────────────────────────┘
```

#### Planning（Sprint Planning / 冲刺计划会）

1. [PM] 读取 `Backlog.md`（含老板便签区），收集新想法
2. [PM] 对模糊想法执行苏格拉底式需求澄清（WHY/WHO/WHAT/HOW）
3. [PM] 将便签条目结构化为 Epic → Story，写入产品待办表
4. [PM] 执行依赖分析和里程碑规划（首次项目时）
5. [Scrum Master] 评估优先级，确认本 Sprint 范围
6. [Scrum Master] 从 Story 拆解为 Task，写入 `任务清单.md`
7. 【汇报】向老板汇报 Sprint 计划，请求确认

#### Execute（Sprint Execution / 冲刺执行）

1. [Backend/Frontend] 检索项目代码，理解现有架构
2. [Backend/Frontend] 逐 Task 执行：编码 → 编译验证 → 标记完成
3. 每个 Task 完成后立即回填 `变更记录.md`
4. 遇到阻塞标记 ❌ 并主动报告

#### Review（Sprint Review / 冲刺评审会）

1. [QA] 确认所有 Task 满足 DoD
2. [QA] 执行编译验证 `go build`
3. [QA] 运行相关单元测试
4. [QA] 检查代码规范合规性

#### Retrospective（Sprint Retrospective / 冲刺回顾会）

1. [Scrum Master] 更新 `Backlog.md` 中对应 Story 的状态
2. [Scrum Master] 检查是否需要归档历史 Sprint
3. [Scrum Master] 识别新增需求或遗留问题，追加到 Backlog
4. 【汇报】向老板提交 Sprint 交付报告

### 6.3 回填规则

| 规则 | 说明 |
| --- | --- |
| 先读后做 | 执行前必须读取 `Backlog.md` 和 `开发计划.md`，理解上下文 |
| 边做边记 | 每完成一个 Task 立即更新 `任务清单.md` 状态 |
| 做完回填 | Task 完成后必须追加 `变更记录.md` |
| 编译验证 | 所有代码变更必须通过 `go build` 验证 |
| 不改设计 | 除非用户明确要求，否则不修改 `开发计划.md` 的核心设计 |
| 及时归档 | Sprint 完成后检查是否触发归档条件 |

### 6.4 Sprint 编号规范

| 格式 | 示例 | 说明 |
| --- | --- | --- |
| S{两位数字} | S01 | 项目内自增，从 S01 开始 |
| 任务编号 | S01-01 | Sprint编号 + 任务序号 |
| Backlog编号 | B-001 | 全局自增，跨 Sprint |

## 八、新项目初始化检查清单

创建新项目目录时，按以下步骤执行：

1. 在 `CP-开发流程/` 下创建以业务名称命名的目录
2. 从 `template/` 复制 4 个模板文件到新目录
3. 根据用户提供的大纲或需求，填充 `开发计划.md`
4. 将需求拆解为 Story 写入 `Backlog.md`
5. 从 Backlog 中选取首个 Sprint 的范围，拆解为 Task 写入 `任务清单.md`
6. 开始执行 Sprint S01
