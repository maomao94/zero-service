# AI 协作指导规则

## 使用方式

- 本文件是跨项目通用 AI 指导规则，优先保持稳定。
- 项目语言、技术栈、业务边界、模块落点、构建命令写在项目级规则文件中。
- 新项目迁移时，通常复用本文件，只改项目级规则文件。

## 修改前流程

1. 判断需求是否属于项目级规则定义的业务范围；超出范围先确认。
2. 判断代码落点：业务实现、API 契约、配置、脚本、数据库、测试分别放到对应模块。
3. 先阅读相邻实现，复用现有分层、命名、返回值、异常处理、工具类和依赖。
4. 确认是否需要改 API/协议、实现模块、SQL/Mapper、配置、依赖或生成代码。
5. 按最小影响范围修改，避免无关 diff 和大范围格式化。
6. 总结改动、影响范围、验证方式；未构建或未测试时说明原因。

## 沟通与提问

- 默认全程中文，包括提问、回答、任务清单和总结；用户明确切换语言时再切换。
- Agent Mode 默认自主推进，能通过搜索代码、阅读配置、相邻实现推断的低风险事项不要频繁打断用户。
- 必须询问的情况：意图多解且影响大、需求可能超出项目边界、不可逆/高风险操作、缺关键参数且无法推断、规则冲突无优先级。
- 提问一次合并相关疑问，通常 1–4 个问题；每题优先给 2–4 个中文选项，并允许“其他”。
- 用户已经回答过的问题不要重复问；已明确的项目边界、技术偏好、构建策略、模块落点要作为后续约束。
- 基于上下文做了未确认假设时，最终回复简要说明假设、影响范围和可调整点。
- 多步骤任务使用任务清单跟踪；回复说明完成内容、修改文件、影响范围、是否验证，不暴露工具内部细节。

## 安全与变更纪律

- 不新增或打印明文密码、Token、密钥、认证头、证书、身份证号、手机号等敏感信息。
- 对外说明需脱敏内网地址、账号、数据库连接、对象存储配置、个人本地路径、仓库路径和 IDE 配置路径等敏感内容。
- 写入规则、文档、提交信息或总结时，不记录个人绝对路径；必要时改写为“用户级配置”“项目内目录”“本地仓库配置”等通用描述。
- 不主动提交代码；只有用户明确要求提交时才执行 git commit。
- 不主动创建额外文档文件，除非用户明确要求或规则文件本身就是任务目标。
- 不主动添加注释，除非用户明确要求；代码注释保持文件既有风格。

## Git 提交说明

- 提交前先查看实际变更内容，提交说明必须结合 diff 总结，不写泛泛模板。
- 提交标题优先中文，简短总结性质，建议不超过一行，例如“优化防火云台列表展示字段”。
- 可使用 conventional commit 前缀，但描述仍尽量中文，例如 `feat: 新增火情任务状态同步`、`fix: 修复机巢任务回调状态判断`。
- 变更较复杂时可在提交正文补 1–3 条关键说明，说明主要改动、影响范围或验证方式；不要写过长流水账。
- 不在提交信息中写敏感信息、内网地址、账号、Token、密钥或完整异常堆栈。

## 迁移建议

- `ai-rule.md`：放通用 AI 行为、提问、安全、提交规则，跨项目复用。
- `project-rule.md`：放项目画像、业务边界、模块落点、编码规则、构建验证，按项目定制。
- 如果项目有技能或 workflow，可放在项目级规则中，避免污染通用 AI 指导规则。
