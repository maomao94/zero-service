## 🎯 二、核心能力

### 必须支持

1. 批量请求执行

    * 支持一次提交多个调用任务

2. 并发执行

    * 所有请求默认并发执行
    * 支持最大并发数控制（防止打爆下游）

3. 多协议支持

    * HTTP 调用 参考 trigger 的 http 回调
    * gRPC 调用（简化版）参考 trigger 的 protogrpc 回调

4. 统一结果返回

    * 每个请求独立返回结果
    * 包含成功 / 失败 / 耗时 （可以结合一下 http grpc 的异同，可能需要转换一下状态码）

5. 超时控制

    * 支持整体请求超时
    * 单个请求失败不影响其他请求

---

## 🧩 三、系统定位

### ✅ 本系统是

* 并发执行器（Execution Engine）
* 请求聚合器（Aggregator）
* 远程 Future 替代方案

---

### ❌ 本系统不是

* 工作流引擎（不做 DAG）
* 调度系统（不做定时任务）
* 队列系统（不做持久化）
* 业务系统（不处理业务逻辑）

---

## ⚙️ 四、执行模型

### 执行方式

* 所有任务默认并发执行
* 执行顺序不保证
* 每个任务独立运行

---

### 并发控制

* 支持 maxConcurrency 参数
* 限制同时执行的任务数量

---

### 超时策略

* 整体请求超时（统一截止） 这个交给 trigger 的服务超时
* 单任务超时由执行器控制 支持自定义单个任务的超时，但是需要判断整体的接口超时
* 超时任务返回失败结果

---

### 错误处理

* 单个任务失败不影响整体
* 返回聚合结果（部分成功 / 部分失败）

---

## 🌐 五、调用类型支持

### HTTP 调用

支持：

* GET / POST 等基础方法
* Header 透传(只是 header 字段透传)
* Body 透传

---

### gRPC 调用（第一版）

* 简化调用方式
* 使用 pb 字节码 （参考 trigger grpc 回调）
* 不处理复杂序列化

---

## 📊 六、返回结果设计

每个请求必须包含：

* 请求唯一标识（id）
* 执行状态（成功 / 失败）
* 错误信息（如有）
* 返回数据
* 执行耗时

## 🧪 七、使用场景

### 典型替代场景

#### 场景1：接口聚合

```text
原来：
接口 A → 调 B → 调 C → 调 D（串行）

现在：
接口 A → 调 Invoke → 并发执行 B C D
```

---

#### 场景2：Java 并发替代

```text
原来：
CompletableFuture + 线程池

现在：
远程调用 Invoke 服务
```

## 🧠 十一、设计原则

1. 极简优先（避免过度设计）
2. 面向替代 Future（而不是构建复杂平台）
3. 无状态服务（便于扩展）
4. 每个请求独立执行
5. 可快速落地（1~2 天可完成 MVP）
6. 结合 antx 框架做流程并发编排

---

## 👍 十二、总结

该服务的本质是：

> 👉 将“本地并发编排（Future）”升级为“远程并发执行能力”

第一阶段目标只有一个：

✔ 简单可用
✔ 可替代 Future
✔ 可直接提升接口性能

而不是：

❌ 功能复杂
❌ 一步做到工作流引擎

### 设计决策记录

1. **HTTP 框架选择**：使用 go-zero `httpc.Service.DoRequest` 而非 `net/http`，复用框架的 circuit breaker 和请求日志
2. **并发编排**：使用 `antsx.InvokeAllSettled` / `InvokeAllSettledWithReactor`，而非手写 WaitGroup + semaphore
3. **gRPC 泛化调用**：trigger 不持有调用方 proto 定义，使用 rawCodec 透传字节码，调用方自行解码
4. **状态码统一**：返回结果的 statusCode 统一为 HTTP 状态码体系，gRPC 错误通过 `gtwx.GrpcCodeToHTTPStatus` 转换
5. **Debug 调试**：gRPC 字节码通过 `protowire` 做 raw decode 输出类 JSON（field number → value），无需 proto schema
