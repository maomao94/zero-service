# 开发规范

本文档定义 `CP-开发计划/{版本号}/` 目录下的文件组织规范和 AI 协作开发流程。
所有版本目录必须遵循此规范，形成统一的「计划 → 执行 → 回填」闭环。

---

## 一、目录结构

每个版本目录（如 `0.0.2/`）应包含以下文件：

```
CP-开发计划/{版本号}/
├── 开发计划.md        # 核心设计文档（目标、边界、接口、架构）
├── 开发规范.md        # 本文件，协作流程与文件约定（全版本共享，放在版本目录外或首个版本中）
├── 任务清单.md        # 当前版本的任务拆解与执行状态跟踪
├── 变更记录.md        # 已完成的变更记录（每轮迭代追加）
└── 下一步计划.md      # 草稿输入文件，用户写入下一步需求，AI 读取后润色并执行
```

---

## 二、各文件职责

### 开发计划.md
- 定义服务的**核心设计**：目标、系统定位、接口设计、SDK 设计、执行模型
- 只记录**设计层面**的内容，不记录执行状态
- 初始版本由用户或 AI 协作产出，后续版本可继承并演进
- 不包含变更记录（变更记录拆到独立文件）

### 任务清单.md
- 当前版本的**任务拆解与执行状态**
- 每个任务有明确的：编号、描述、状态（⬜ 未开始 / 🚧 进行中 / ✅ 完成）、执行说明
- AI 执行完任务后必须立即回填状态
- 版本结束后任务清单归档，不再修改

### 变更记录.md
- 记录每轮迭代的**已完成变更**
- 按时间倒序排列，最新的在最前面
- 每条记录包含：变更概述、具体内容、影响范围
- AI 每完成一轮工作必须追加变更记录

### 下一步计划.md
- **用户的草稿输入文件**
- 用户可以直接在此文件中写入简要的需求、想法、待办事项
- AI 读取后：
  1. 润色需求描述，补充技术细节
  2. 拆解为具体任务写入 `任务清单.md`
  3. 开始执行
  4. 执行完毕后回填 `变更记录.md` 和 `任务清单.md`
  5. 清空或标记 `下一步计划.md` 中已处理的条目

---

## 三、AI 协作流程

```
用户写入 下一步计划.md
        ↓
AI 读取并润色需求
        ↓
AI 拆解任务 → 写入 任务清单.md
        ↓
AI 逐项执行（编码、测试、编译验证）
        ↓
AI 回填 任务清单.md（标记完成）
        ↓
AI 追加 变更记录.md（记录本轮变更）
        ↓
AI 更新 下一步计划.md（标记已处理）
```

### 执行规则

1. **先读后做**：AI 在执行前必须先读取 `下一步计划.md` 和 `开发计划.md`，理解上下文
2. **边做边记**：每完成一个任务立即更新 `任务清单.md` 状态
3. **做完回填**：一轮工作结束后必须追加 `变更记录.md`
4. **不改设计**：除非用户明确要求，否则不修改 `开发计划.md` 的核心设计部分
5. **编译验证**：所有代码变更必须通过 `go build` 验证

---

## 四、命名约定

| 项目 | 约定 |
| -- | -- |
| 版本号 | 语义化版本 `x.y.z`，如 `0.0.1`、`0.0.2` |
| 文件命名 | 中文命名，`.md` 格式 |
| 任务编号 | 版本内自增，如 `#1`、`#2` |
| 任务状态 | ⬜ 未开始 / 🚧 进行中 / ✅ 完成 |
| 变更记录 | 按日期或轮次编号 |

---

## 五、版本迭代规则

1. 新版本继承上一版本的 `开发计划.md`（可修改演进）
2. `任务清单.md`、`变更记录.md`、`下一步计划.md` 每个版本独立
3. 版本结束后整个目录归档，不再修改
4. 开发规范文件可放在 `CP-开发计划/` 根目录下全版本共享
