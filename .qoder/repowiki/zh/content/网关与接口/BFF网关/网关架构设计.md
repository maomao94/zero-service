## 引言
本文件面向BFF（Backend For Frontend）网关的架构设计与实现，围绕RESTful API服务器初始化、服务上下文（ServiceContext）设计与依赖注入、配置文件加载与解析、CORS跨域处理、服务注册与启动流程、日志系统集成与全局字段添加、以及新增的全局中间件认证类型设置等方面进行系统化梳理，并通过多种可视化图表帮助开发者快速理解网关内部工作原理。

## 项目结构
网关模块位于多个目录中，采用"入口程序 + 配置 + 服务上下文 + 路由注册"的分层组织方式：
- 入口程序负责加载配置、初始化REST服务器、注册路由、启动服务组
- 配置模块定义运行时配置结构体，包含Host、Port、SwaggerPath、JwtAuth、zrpc客户端配置等
- 服务上下文聚合业务所需依赖（校验器、RPC客户端、支付SDK等），作为处理器的共享容器
- 路由注册模块集中声明各业务前缀下的REST路由及鉴权策略
- 中间件与上下文数据模块负责在gRPC调用中传递用户与链路追踪信息
- **新增** 全局中间件负责设置认证类型为'user'，确保来自浏览器的请求能够正确识别为用户级认证

### 全局中间件认证类型设置
**新增** 所有网关服务（AI网关、通用网关、Socket网关）都增加了全局中间件来设置认证类型为'user'，确保来自浏览器的请求能够正确识别为用户级认证。

- **实现机制**：在REST服务器初始化后，立即注册全局中间件，通过`context.WithValue`将认证类型键值对注入到请求上下文中
- **认证类型键**：使用`ctxdata.CtxAuthTypeKey`常量，值为"user"，标识请求来自浏览器的用户认证
- **Authorization头处理**：如果请求包含Authorization头，同时将原始令牌注入到上下文中，供后续gRPC拦截器使用
- **执行时机**：全局中间件在路由匹配之前执行，确保所有请求都经过认证类型标记

```mermaid
flowchart TD
Start(["请求到达网关"]) --> SetAuthType["设置认证类型<br/>ctxdata.CtxAuthTypeKey = 'user'"]
SetAuthType --> CheckAuth{"是否存在Authorization头?"}
CheckAuth --> |是| SetAuthHeader["设置Authorization头<br/>ctxdata.CtxAuthorizationKey = 原始令牌"]
CheckAuth --> |否| SkipAuth["跳过Authorization头设置"]
SetAuthHeader --> Next["继续中间件链"]
SkipAuth --> Next
Next --> RouteMatch["路由匹配与处理"]
```

### REST服务器初始化与CORS
- 初始化方式：使用REST框架提供的服务器工厂函数，传入运行配置并启用自定义CORS回调
- CORS策略要点：
  - 动态Origin：根据请求头中的Origin设置响应头，避免固定域名限制
  - 预检请求支持：明确允许OPTIONS方法，确保浏览器预检正常
  - 凭证与头部白名单：允许携带Cookie/Token，开放常用请求头与暴露响应头
  - Vary头：避免缓存污染，按Origin区分缓存
- Swagger静态路由：当配置中存在SwaggerPath时，动态注册/swagger/:fileName路由，用于暴露本地JSON文件

```mermaid
flowchart TD
Start(["进入CORS回调"]) --> CheckOrigin["读取请求头 Origin"]
CheckOrigin --> HasOrigin{"Origin 是否为空?"}
HasOrigin --> |是| SetAllowOriginEmpty["设置允许的Origin为空"]
HasOrigin --> |否| SetAllowOrigin["设置允许的Origin为请求Origin"]
SetAllowOriginEmpty --> SetVary["设置 Vary: Origin"]
SetAllowOrigin --> SetVary
SetVary --> SetCredentials["允许凭据: true"]
SetCredentials --> SetMethods["允许方法: GET,POST,PUT,DELETE,OPTIONS,PATCH"]
SetMethods --> SetHeaders["允许头部: 多个常用字段"]
SetHeaders --> ExposeHeaders["暴露响应头: Content-Length, Content-Type"]
ExposeHeaders --> End(["完成CORS设置"])
```

### JWT认证映射机制
**新增** 当配置中存在JwtAuth.ClaimMapping时，网关会注册第二个全局中间件来处理JWT声明映射。

- **映射原理**：将外部JWT声明键（如"user_id"）映射为内部标准键（如"user-id"），确保下游服务能够正确识别用户信息
- **执行时机**：该中间件在go-zero JWT中间件之后执行，此时JWT claims已经注入到上下文中
- **实现方式**：通过`ctxprop.ApplyClaimMappingToCtx`函数读取外部键值，写入内部标准键
- **应用场景**：当上游系统使用不同的JWT声明命名规范时，通过此机制实现兼容

```mermaid
flowchart TD
Start(["JWT认证完成"]) --> CheckMapping{"是否存在ClaimMapping?"}
CheckMapping --> |是| ApplyMapping["应用声明映射<br/>ApplyClaimMappingToCtx"]
CheckMapping --> |否| SkipMapping["跳过映射处理"]
ApplyMapping --> ReadExternal["读取外部声明键值"]
ReadExternal --> WriteInternal["写入内部标准键"]
WriteInternal --> Next["继续处理流程"]
SkipMapping --> Next
```

## 性能考虑
- 路由超时：针对文件上传等长耗时操作设置独立超时，避免阻塞其他请求
- 并发控制：RPC客户端与任务执行器具备并发与限流能力，建议结合实际QPS调整
- CORS成本：动态Origin设置与Vary头会增加少量CPU开销，通常可忽略
- **新增** 全局中间件性能：认证类型设置中间件为O(1)操作，对性能影响极小
- 日志输出：生产环境建议使用异步日志与滚动切分，避免I/O成为瓶颈

## 故障排查指南
- CORS相关问题
  - 现象：浏览器报跨域错误或预检失败
  - 排查：确认请求头Origin是否正确传递，响应头是否包含允许的方法与头部
  - 参考：CORS回调设置的允许方法、头部与凭证策略

  - 排查：确认全局中间件是否正确执行，检查`ctxdata.CtxAuthTypeKey`是否设置为"user"
  - 解决：验证中间件注册顺序，确保在路由处理之前执行
- **新增** JWT声明映射问题
  - 玔象：用户信息字段缺失或格式不正确
  - 排查：确认JwtAuth.ClaimMapping配置是否正确，检查外部JWT声明键名
  - 解决：验证声明映射配置，确保内部标准键与下游服务期望一致
- Swagger静态文件无法访问
  - 现象：访问/swagger/:fileName返回404
  - 排查：确认配置文件中SwaggerPath指向的目录存在且可读
- JWT鉴权失败
  - 现象：用户相关接口返回未授权
  - 排查：核对JWT签名密钥与前端令牌生成一致，检查路由是否启用JWT中间件
- RPC调用异常
  - 现象：下游服务不可达或响应超时
  - 排查：检查RPC客户端配置、网络连通性与下游健康状态；确认元数据拦截器已注入
